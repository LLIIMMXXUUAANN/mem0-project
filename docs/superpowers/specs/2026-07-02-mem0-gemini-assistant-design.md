# Multi-User Mem0 + Gemini Personal Assistant — Design

Date: 2026-07-02

## Purpose

A learning project to explore mem0's memory API (add/search/get_all/delete) in a
realistic multi-user setting. A Streamlit chat app lets you converse as different
users; each user's facts/preferences are stored and retrieved via **Mem0 Cloud**,
and visible both in the app's own memory browser panel and in mem0's hosted
dashboard (app.mem0.ai). Chat replies are generated by **Gemini 2.5 Flash**.

## Scope decisions

- **Mem0 mode: Mem0 Cloud** (hosted platform, API key from app.mem0.ai). Memory
  extraction (deciding what facts to save from a conversation) is handled by
  Mem0's own managed backend — this is *not* swappable to Gemini. Gemini powers
  the assistant's chat replies only, not the internal memory-extraction step.
  This tradeoff was explicitly discussed and accepted in favor of getting the
  hosted per-user dashboard.
- **Interface:** Streamlit web UI (not a CLI).
- **Multi-user:** free-text `user_id` entry in the sidebar — no auth, no backend
  user registry. Switching the field switches which user's memories are used.
- **Assistant scope:** chat with automatic memory (mem0 auto-extracts facts each
  turn) plus a dedicated memory browser panel (list/search/delete) so the mem0
  API surface is directly visible, not just implicit in the chat loop.
- **Transparency:** each assistant reply shows an expandable "memories used"
  section listing exactly what mem0 retrieved for that turn.
- **API keys:** user already has both a Mem0 API key and a Gemini API key; no
  onboarding/signup instructions needed, just where to put them (`.env`).

## Architecture

Python project managed with `uv`. Dependencies: `streamlit`, `mem0ai`,
`google-genai`, `python-dotenv`.

```
mem0-project/
├── app.py             # Streamlit UI: chat + memory browser panel
├── mem0_service.py    # thin wrapper around mem0 MemoryClient (cloud)
├── gemini_service.py  # thin wrapper around Gemini 2.5 Flash chat call
├── pyproject.toml     # uv-managed deps
├── .env.example       # MEM0_API_KEY=, GEMINI_API_KEY=
└── README.md          # setup + manual smoke-test checklist
```

### Components

- **`mem0_service.py`** — all mem0 calls funnel through here:
  - `add_memory(user_id, messages)`
  - `search_memories(user_id, query, limit)`
  - `get_all_memories(user_id)`
  - `delete_memory(memory_id)`
- **`gemini_service.py`** — the only place Gemini is called:
  - `generate_reply(user_message, memory_context, chat_history) -> str`
    Builds a system prompt injecting the retrieved memories, calls Gemini
    2.5 Flash, returns the reply text.
- **`app.py`** — Streamlit UI:
  - Sidebar: `user_id` text input.
  - Main area: chat (`st.chat_input` / `st.chat_message`), with an expandable
    "memories used" box under each assistant reply.
  - Memory browser panel (separate section/tab): lists all memories for the
    current user (`get_all_memories`), a search box (`search_memories`), and
    delete buttons (`delete_memory`).

## Data flow

1. User types/switches `user_id` in the sidebar → on-screen chat history in
   `st.session_state` resets for that user. (mem0's stored memories are *not*
   reset by this — they persist server-side keyed by `user_id`; this only
   clears the visible transcript.)
2. User sends a chat message.
3. `app.py` calls `mem0_service.search_memories(user_id, message)` → relevant
   memory strings for this user.
4. `app.py` calls `gemini_service.generate_reply(message, memories, chat_history)`
   → Gemini 2.5 Flash produces a reply informed by those memories.
5. Reply is shown in the chat, with the "memories used" box listing exactly
   what was retrieved in step 3.
6. `app.py` calls `mem0_service.add_memory(user_id, [user_msg, assistant_reply])`
   → mem0 Cloud auto-extracts and stores any new durable facts in the
   background.
7. Independently, the memory browser panel calls `get_all_memories(user_id)` on
   demand (button/refresh) to list everything stored for that user, with
   search (`search_memories`) and delete (`delete_memory`) actions — letting
   the mem0 state be inspected and manipulated outside the chat flow.

## Error handling

- On startup, `app.py` checks both API keys are present; if missing, shows a
  clear `st.error` with instructions instead of crashing mid-session.
- Both service modules catch exceptions from their respective SDKs (network
  errors, rate limits, bad requests) and raise a small typed exception with a
  friendly message; `app.py` catches these at the call site and shows
  `st.error(...)` instead of crashing the app.
- If a turn fails after the Gemini call but before `add_memory`, that turn is
  not stored (no partial/broken memory writes).
- Empty search results are a normal case (new user, no memories yet) — an
  empty list is passed into the prompt, not treated as an error.

## Testing

This project's point is exercising two live external APIs (mem0 Cloud,
Gemini), so heavy mocked unit tests would mostly test the mocks rather than
provide learning value. Instead:

- A manual smoke-test checklist in `README.md`: create user "alice", tell the
  assistant a fact, switch to user "bob", confirm bob's assistant doesn't know
  it, switch back to alice and confirm it remembers, exercise the memory
  browser's search/delete.
- If any pure logic emerges worth unit testing (e.g. prompt-building in
  `gemini_service.py`), a small `test_gemini_service.py` with no network calls
  is acceptable — opportunistic, not a requirement.

## Out of scope

- Mem0 OSS / self-hosted mode.
- Authentication / real user accounts.
- Persisting chat transcripts locally (only mem0's extracted facts persist;
  the visible chat log is per Streamlit session).
- Automated test suite covering the external API integrations.
